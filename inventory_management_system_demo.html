Why Redis for Rate Limiting?
Redis stores each user's request count as a key:value pair with an atomic INCR and a sliding TTL. Operations are O(1) and sub-millisecond — a relational DB would need transactions and row-locks, adding latency every request. The in-memory store handles thousands of concurrent counters without I/O overhead. Each key auto-expires after the window (60s), so no cleanup jobs needed.

Why MongoDB for Inventory?
Inventory items are heterogeneous — a laptop has RAM/GPU fields; furniture has dimensions/material. MongoDB's schema-less documents let each product carry exactly the fields it needs without NULL columns. A stack (LIFO) is modelled via push/pop on an array field, ideal for warehouse intake logs. Horizontal sharding handles catalog growth without schema migrations.

Why Neo4j for Product–Manufacturer Correlation?
Supplier chains and product relationships are inherently graph-shaped: a product can have multiple manufacturers, suppliers can supply multiple products, and finding indirect relationships (e.g. "which products share a supplier?") requires multi-hop traversal. In SQL this needs costly multi-join queries. Neo4j stores nodes and edges natively — traversal is pointer-chasing O(depth), not O(rows×joins). Cypher queries express "find all manufacturers 2 hops away" in one readable line.
